# Internationalization & Message Catalog — Binding

Language switching is a **premise** of NeNe Field, not an add-on. **Every
user-facing string lives in a per-locale message catalog file**, addressed by a
stable key, so the UI can switch language **at runtime without a reload** and new
locales drop in by adding one catalog.

> **Status: binding.** A hardcoded user-facing string, a missing catalog key, or
> displaying raw API text to users **blocks merge to `main`**. This operationalizes
> [ADR 0012](../adr/0012-international-readiness.md) (i18n-ready, `en` first-class)
> and is recorded in [ADR 0015](../adr/0015-message-catalog-and-locale-switching.md).
> Adopts the sibling **nene-invoice** i18n implementation.

**Read with:** [`frontend-standards.md`](./frontend-standards.md),
[`../terms.md`](../terms.md) §7 (Problem Details slugs). Self-review:
[`../review/frontend.md`](../review/frontend.md).

---

## 1. Principle

1. **One catalog file per locale.** `messages/ja.ts` and `messages/en.ts` are the
   message-list files; **`ja.ts` is the authoritative master**.
2. **No hardcoded user-facing strings.** Components render copy only through
   `t('key')` — never string literals, never concatenated sentence fragments.
3. **Seamless runtime switching.** Changing locale re-renders from the active
   catalog with **no page reload**, and the choice is persisted.
4. **Type-safe keys.** A key that is not in the master catalog is a **compile
   error**; this is the same zero-typo discipline as `terms.md`, applied to UI copy.
5. **Locale ⊇ parity.** Every key in `ja` MUST exist in `en` (and vice versa) —
   enforced in CI. Missing keys fall back to `ja` at runtime, never to a blank.

---

## 2. Catalog structure (`shared/i18n/`)

```text
shared/i18n/
  locales.ts          # SupportedLocale, DEFAULT_LOCALE, LOCALES, resolveLocale()
  translate.ts        # MessageKey, MessageCatalog, pure translate(catalog,key,params)
  messages/
    ja.ts             # AUTHORITATIVE master catalog (every key defined here)
    en.ts             # Partial<…>; falls back to ja for any missing key
  context.ts          # I18nContext + I18nContextValue type
  i18n-context.tsx    # I18nProvider — detect / persist / switch
  use-translation.ts  # useTranslation() → { locale, setLocale, t }
  index.ts            # public barrel
```

Type model (the master catalog defines the key universe):

```ts
// translate.ts
export type MessageKey = keyof typeof jaMessages          // unknown key ⇒ compile error
export type MessageCatalog = Partial<Record<MessageKey, string>>
export type TranslateParams = Record<string, string | number>

// translate() resolves active catalog → falls back to jaMessages → interpolates {{name}}
```

- `t(key, params)` accepts **only** a `MessageKey`. There is no untyped string
  lookup and no dynamic key built by concatenation.
- Interpolation uses `{{name}}` placeholders, never string concatenation of
  translated fragments (word order differs across languages).

## 3. Key naming

Dot-namespaced, lowercase, mirroring screens/features:

| Scope | Pattern | Example |
| --- | --- | --- |
| Shared chrome | `common.{group}.{element}` | `common.actions.save`, `common.error.generic`, `common.locale.ja` |
| Screen / feature | `{feature}.{element}` | `report.submit.title`, `report.list.empty`, `auth.login.button` |
| Server error message | `error.{problem-slug}` | `error.report-not-found`, `error.forbidden` |
| Validation message | `error.validation.{code}` | `error.validation.required` |

Feature/entity names match the frontend FSD slices and the OpenAPI tags. Adding a
string means adding the key to **both** `ja.ts` and `en.ts` in the same PR.

## 4. Runtime switching, detection & persistence

`I18nProvider` (in `app/providers.tsx`) holds the active locale and exposes
`{ locale, setLocale, t }` via `useTranslation()`:

- **Switch:** `setLocale('en')` updates context state → React re-renders from the
  new catalog **without a reload**.
- **Persist:** the choice is saved to `localStorage['nene-field-locale']`
  (best-effort).
- **Detect (first load):** `localStorage` → `navigator.language` → `DEFAULT_LOCALE`
  (`ja`). `resolveLocale()` maps any input to a `SupportedLocale`.
- A **locale switcher** lives in the app shell (e.g. account menu); its labels use
  `common.locale.*` keys.

`SupportedLocale = 'ja' | 'en'`, `DEFAULT_LOCALE = 'ja'`. A third locale requires an
ADR (ADR 0012) and a full `en`-parity catalog before merge.

## 5. Server messages localize on the client (the bridge)

The API stays **English with stable machine identifiers** (ADR 0006): Problem
Details `type`/slug (`terms.md §7`), validation `code`, and HTTP `status`. The UI
**maps the slug/code to a catalog key** and renders the localized message:

```
API: 404 application/problem+json  type=…/problems/report-not-found
  → shared/api/errors.ts derives slug "report-not-found"
  → UI shows t('error.report-not-found')         // localized
```

- The UI **MUST NOT** display the raw API `title`/`detail` to end users; those are
  English diagnostics. Features map `slug`/`status`/validation `code` → catalog.
- **Every Problem Details slug in `terms.md §7` and every validation `code` MUST
  have a catalog message** (`error.*`). A new slug without a catalog entry blocks merge.

---

## 6. Locale-aware formatting (not translation)

- Dates, times, and numbers are formatted with `Intl` using the **active locale**;
  never hardcode a format or timezone in components. Instants are stored UTC
  (ADR 0011); JST is the Japan-edition default display (ADR 0012).
- This is presentation only — the stored data is unchanged.

---

## 7. What is NOT in the catalog

- **User data is content, not UI copy** — report `title`/`body`, tags, user names,
  template field values are never translated and never go through `t()`.
- **AI-generated summaries** are content (the user's data), not UI strings.
- **Japan-pack statutory text** (Japan edition) is governed by `legal-compliance.md`,
  not the UI catalog.

---

## 8. Testing (binding)

- **Key parity test:** assert `en` defines every `MessageKey` in `ja` (and no extra
  keys). This is mandatory — it is the guard that prevents the "ja 625 / en 623"
  drift seen in a sibling product.
- **`translate()` unit tests:** active-catalog hit, ja fallback for a missing en key,
  `{{param}}` interpolation, missing-param passthrough.
- **No-hardcoded-string lint:** ESLint flags literal user-facing JSX text outside the
  catalog where feasible; reviewers reject the rest.

---

## 9. Forbidden (blocks merge)

- A hardcoded user-facing string (literal JSX text, `alert`/button/label text) not
  routed through `t()`.
- A dynamic/concatenated message key, or concatenating translated fragments into a
  sentence (use `{{param}}` interpolation).
- Adding a key to `ja.ts` without `en.ts` (or vice versa) — fails key parity.
- Displaying raw API `title`/`detail` to users instead of mapping slug/code → catalog.
- A Problem Details slug (`terms.md §7`) or validation code with no `error.*` catalog
  message.
- A third locale without an ADR and a complete parity catalog.
- Running `t()` over user data (report body, names, tags).
